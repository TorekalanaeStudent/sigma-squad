# 0. MVP SCOPE (CRITICAL DECISION)

### MVP Goal

Deliver a working system where:

* Students can login
* Students can see computer availability
* Students can reserve 1 computer
* Reservation expires after 5 minutes if not confirmed
* Librarian can assign / end sessions
* System prevents double booking

---

# 1. DOMAIN MODEL (CORE ENTITIES)

## 1.1 User

```java
User
- id (UUID)
- name (String)
- studentId (String, unique)
- email (String, unique)
- passwordHash (String)
- role (ENUM: STUDENT, LIBRARIAN)
- createdAt (Instant)
```

### Rules

* Email must end in `@nu.edu.ph` (or your institution domain)
* studentId required for STUDENT only
* password stored hashed (BCrypt)

---

## 1.2 Computer

```java
Computer
- id (UUID)
- computerNumber (int, unique)
- status (ENUM: AVAILABLE, RESERVED, IN_USE, OUT_OF_SERVICE)
- currentUserId (UUID, nullable)
```

### Rules

* One computer can only be assigned to one active user/session
* Status is derived from Reservation/Session but stored for fast access (MVP simplification)

---

## 1.3 Reservation

```java
Reservation
- id (UUID)
- userId (UUID)
- computerId (UUID)
- status (ENUM: ACTIVE, EXPIRED, CANCELLED, CONFIRMED)
- reservedAt (Instant)
- expiresAt (Instant) // reservedAt + 5 minutes
```

### Rules

* One ACTIVE reservation per user only
* Auto-expire after 5 minutes if not confirmed

---

## 1.4 Session

```java
Session
- id (UUID)
- userId (UUID)
- computerId (UUID)
- startTime (Instant)
- endTime (Instant, nullable)
- status (ENUM: ACTIVE, ENDED)
```

# 4. BUSINESS LOGIC (CRITICAL RULES)

## 4.1 Reservation Rules

* Max 1 active reservation per user
* If user creates reservation:

  * Computer → RESERVED
  * Reservation expires in 5 minutes

---

## 4.2 Expiration Logic (VERY IMPORTANT)

### Spring Scheduler (every 30–60 seconds)

```java
@Scheduled(fixedRate = 60000)
```

Checks:

* If `now > expiresAt`
* Set reservation → EXPIRED
* Set computer → AVAILABLE

---

## 4.3 Confirmation Flow

1. Student reserves computer
2. Librarian confirms
3. Reservation → CONFIRMED
4. Session created
5. Computer → IN_USE

---

# 9. EDGE CASES (MUST HANDLE)

## Reservation

* User refreshes page → reservation still active
* User tries double reservation → block
* Computer already reserved → reject

---

## Session

* Librarian ends session early → computer becomes AVAILABLE
* Session ends → clear currentUserId

---

## Expiration

* Scheduler must NOT double-expire
* Expired reservation cannot be confirmed

---
